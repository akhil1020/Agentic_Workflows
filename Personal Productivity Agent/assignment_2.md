# Assignment 2:  

## Overview

In this assignment you will build a **Personal Productivity Agent** using AISuite and tool calling. The agent will help a user manage their day by checking the weather, reading and writing notes, setting reminders, and generating a shareable QR code — all through natural language prompts.

---

## Scenario

You are building a morning briefing agent for a busy professional. Every morning they ask it one prompt and it:

1. Fetches the current time and weather for their location
2. Reads their existing to-do list from a file
3. Appends a new task to the to-do list
4. Generates a QR code linking to a meeting URL so they can join from their phone

---

## Requirements

### Tools to Build

You must define **4 Python functions** as tools. Each function must have a clear docstring — AISuite uses it to describe the tool to the LLM.

| Tool                     | What it does                                                                      | Parameters                                 |
| ------------------------ | --------------------------------------------------------------------------------- | ------------------------------------------ |
| `get_current_datetime` | Returns the current date and time as a formatted string                           | None                                       |
| `get_weather_summary`  | Detects location from IP and returns current temp, daily high, and low in Celsius | None                                       |
| `manage_todo_list`     | Reads or appends tasks to a local `todo.txt` file                               | `action` (read/add), `task` (optional) |
| `generate_meeting_qr`  | Generates a QR code PNG from a meeting URL                                        | `url`, `filename`                      |

### What You Must Implement

**Part 1 — Automatic tool calling**

Pass all four tools to the LLM with `max_turns` enabled and test each tool with an individual prompt. The LLM should automatically pick the right tool based on the prompt.

**Part 2 — Manual tool call handling**

Define the schema for `manage_todo_list` manually using the JSON schema format. Then handle the tool call loop yourself — without `max_turns` — by extracting the tool name and arguments from the response, running the function locally, appending the result back to the messages list, and making the second API call for the final response. Use a tool dispatcher (a dictionary mapping tool names to functions) instead of hardcoding which function to call.

**Part 3 — Multi-tool morning briefing**

Write a single natural language prompt that causes the LLM to use at least 3 tools in sequence. After running it, inspect and explain the order in which the tools were called.

**Part 4 — Reflection**

Answer these questions in your notebook:

1. Remove the docstring from one of your tools and observe what happens. What changes in LLM behavior?
2. What is the advantage of a tool dispatcher over hardcoding the function call?
3. What would happen if the LLM needed to call a second tool after receiving the first result — how would you extend your manual loop to handle that?
4. The `manage_todo_list` tool writes to the local filesystem. What could go wrong if the LLM were allowed to pass any value without validation?

## Hints

- The Open-Meteo weather API is free and requires no API key.
- For the todo list tool, use append mode `'a'` for writing and read mode `'r'` for reading.
- If the LLM is not picking the right tool, your docstring description is likely too vague — it is the only signal the LLM has.
- Always check whether `tool_calls` exists in the response before trying to access it in manual mode.
